# Remaining backend domains — apply the template, don't re-derive it

`packages/organization.md` is the canonical worked example (simple CRUD domain, no external system). `packages/gallery.md` adds the Saga pattern (two-system operation). `packages/logos.md`/`photos.md`/`icons.md` add the integration-port pattern (external HTTP API instead of a DB). Every remaining domain in `apps/api/src/domains/` is _some combination of those three shapes_ — this doc gives the mechanical template plus per-domain notes on which shape applies, grounded in each domain's actual current file list (confirmed, not guessed). None of these domains' `service.ts` internals were read in depth this session — read the real file before writing the use-case classes; the notes below tell you what to expect, not what to copy verbatim.

## The template, restated as a checklist

For a domain named `<domain>`:

1. `packages/<domain>/package.json` — deps limited to whatever it actually needs (`@deck-pack/db` for DB-backed domains, `@deck-pack/integrations` for external-API domains, `@deck-pack/storage` if it touches blobs, other domain packages **never** as a dependency unless explicitly justified — see the cross-domain note below).
2. `src/domain/` — entity/value object (rich only if there's real behavior/invariants to hang on it, otherwise a plain typed data holder) + `errors.ts` (one `AppError` subclass per current `serviceFail(reason, ...)` tag).
3. `src/repositories/` (DB-backed) and/or `src/integrations/` (external-API-backed) — one interface, one production implementation, one `InMemory*` fake.
4. `src/use-cases/` — one class per current service method, constructor-injected with the repository/integration/port(s) it needs.
5. `src/index.ts` — export use-cases, entity types, the repository/port interface (type-only), error classes.
6. `apps/api/src/routers/<domain>-router.ts` — thin tRPC adapter, replaces `domains/<domain>/routes.ts`.
7. Add the new field(s) to `AppContainer`'s three factories.
8. Delete `apps/api/src/domains/<domain>/`.

## The recurring open question: `gallery`/`flags`/`shapes`/`slides` overlap

Four domains (`gallery`, `flags`, `shapes`, `slides`) all sound related to "the catalog of visual assets a user can insert." Before extracting `flags`, `shapes`, and `slides` as independent packages, explicitly check whether they're:

- (a) genuinely separate bounded contexts that happen to also appear as `GalleryAssetClass` values, or
- (b) thin, asset-class-specific views/queries over the same underlying `gallery_items` table that `packages/gallery` already owns.

If (b), the right move is fewer new packages than this table implies — e.g. `ListFlags` becomes a use-case _inside_ `packages/gallery` (`list-gallery-items.ts` filtered by `assetClass: "flag"`) rather than a whole separate `packages/flags` with its own Repository hitting the same table. This is exactly the kind of premature-package smell `00-conventions-and-architecture.md` §8.3 warns about — don't create a new top-level package until you've confirmed the code doesn't just belong as a sibling inside an existing one. Resolve this with a quick read of `flags/service.ts`, `shapes/service.ts`, `slides/service.ts`, and `gallery`'s (`library/service.ts`) query dependencies before writing any of these four packages.

## Migration order for this batch

1. Read each domain's real `service.ts` (this doc deliberately didn't guess at implementation).
2. Resolve the `gallery`/`flags`/`shapes`/`slides` overlap question first — it changes how many packages you actually create.
3. Migrate `users` and `seats` early (simple, DB-only, low risk).
4. Migrate `members`, checking for the auth-system Saga candidate noted above.
5. Migrate `billing` last among these — highest complexity (likely external API + Saga).
6. Decide `system`/`assets`/`addin` case-by-case per the "probably not a domain" notes — don't force the package-per-domain ceremony onto genuinely small, non-business-logic router files.
