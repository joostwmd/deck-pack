# Assets architecture — views, controllers, services

This document describes the strict view/controller/services pattern used in the Deck Assets add-in (`apps/addins/assets`). Agents should follow these conventions when adding or modifying UI.

## Layer responsibilities

| Layer | File pattern | Responsibility |
| --- | --- | --- |
| **View** | `*-view.tsx` | Pure presentation. Props in, JSX out. No tRPC, Office.js, auth, router, or service hooks. |
| **Controller** | `use-*-controller.ts` | State, side effects, hotkeys, toasts. Composes injectable hooks and reads shared clients from services. |
| **Container** | existing feature file | Thin glue: `const props = useXController(...); return <XView {...props} />;` |
| **Services** | `services/` | Cross-cutting clients: API, auth, Office, insertion, shortcut persistence. |

```text
ServicesProvider (app + Storybook)
        │
        ▼
   Container ──► Controller hook ──► View
        │              │
        │              └── feature fns injected as params (search, getDetails, …)
        └── useServices() for shared clients
```

## Dependency injection

Production wiring lives in `services/app-services.ts` and is mounted from `routes/__root.tsx`.

Storybook and tests use the same interface via `createTestServices()`:

- `fixtures/test-services.ts` — default mocked `AppServices`
- `fixtures/asset-search.ts` — shared mock search data and handlers
- `packages/ui/stories/assets/decorators.tsx` — `StoryProviders` wraps `ServicesProvider`
- `withTestServices(services?)` — decorator for stories that need custom API behaviour

Feature-specific network calls stay **parameterized** on controllers (for example `search` and `getDetails` on asset pickers). Shared infrastructure (insertion strategy, shortcut store, Office client) comes from **`useServices()`**.

## Storybook conventions

Stories live under `packages/ui/stories/assets/` and import add-in code through the `@/` Vite alias.

### View stories (preferred for MCP manifest)

1. Import the **view** component directly.
2. Pass static fixture props from `@fixtures/view-story-fixtures`.
3. Use `withAssetsPanel` for theme, shortcuts, and panel sizing.
4. Add `tags: ['autodocs']` on the meta object.
5. Document each primary story with a JSDoc `@summary` comment.

### Connected container stories

Use for end-to-end flows that prove services wiring (for example `FlagsPanel` with mock tRPC handlers):

- `Default` and interaction stories (`SelectCountry`) are manifest-friendly.
- Edge cases (`NoResults`, `SearchError`, loading, validation errors) use `tags: ['!manifest']`.

### Fixtures

All shared fixtures live at the repo root in `fixtures/` and are imported via the `@fixtures` alias:

- `@fixtures/asset-search` — mock search results and handlers
- `@fixtures/panel-test-services` — panel-specific `AppServices` overrides
- `@fixtures/view-story-fixtures` — view/controller prop builders
- `@fixtures/test-services` — `createTestServices()` factory

Unit tests for the add-in live in `tests/addins/assets/` (mirroring `src/` structure).

## Adding a new feature panel

1. Define types and pure subcomponents first.
2. Extract `feature-view.tsx` with explicit props (including shortcut tokens/defs resolved by the controller).
3. Add `use-feature-controller.ts` — inject feature functions, read services for shared clients.
4. Keep the exported panel as a ~10-line container.
5. Add view stories with fixture props plus one connected story if the flow depends on services.
6. Add hook/controller tests with `createTestServices()` when behaviour is non-trivial.

## Testing hooks and controllers

Wrap `renderHook` with `ServicesProvider` and pass `createTestServices(overrides)` for API or Office behaviour.

Prefer testing controllers and injectable hooks over snapshotting views. Views should remain deterministic given fixture props.

## Agent checklist

- [ ] View file has zero imports from `@/utils/trpc`, `@/utils/auth`, or Office.js
- [ ] Controller receives feature-specific functions as parameters where applicable
- [ ] Container is the only exported component used by routes
- [ ] Storybook view story exists with `autodocs` and `@summary`
- [ ] Error/empty/loading stories tagged `!manifest`
- [ ] Fixtures imported from `@fixtures/*` rather than duplicated in Storybook
